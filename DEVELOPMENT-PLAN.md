# BuyGo+1 完整開發計畫

> **目標**：結束無止盡的重複修復，建立可持續的開發流程
>
> **原則**：先穩定、再發布、後重構
>
> **建立日期**：2026-01-23

---

## 📋 計畫總覽

```
【第 1 階段】穩定期（1 月 24 日 - 1 月 28 日）5 天
├─ 目標：穩定目前版本，修復關鍵 bug
├─ 每天投入：2-3 小時
└─ 產出：可發布的穩定版本

【第 2 階段】發布期（1 月 29 日 - 1 月 31 日）3 天
├─ 目標：部署到正式環境，客戶開始使用
├─ 每天投入：1-2 小時
└─ 產出：線上運行的系統

【休息期】（2 月 1 日 - 2 月 7 日）1 週
├─ 目標：休息，處理其他事情
├─ 每天投入：0 小時（只處理緊急 bug）
└─ 產出：恢復精神

【第 3 階段】重構期（2 月 8 日 - 3 月 7 日）4 週
├─ 目標：模組化重構，從根本解決重複問題
├─ 每週投入：10-15 小時（每天 2-3 小時）
└─ 產出：乾淨、模組化的代碼架構

【第 4 階段】新功能期（3 月 8 日起）
├─ 目標：在新架構上開發多樣式等新功能
├─ 每週投入：10-15 小時
└─ 產出：新功能穩定上線
```

---

## 🎯 第 1 階段：穩定期（5 天）

讓目前的版本「能用」，不追求完美。

1. 打開系統，進入每個頁面
2. 記錄任何明顯的錯誤（頁面空白、按鈕無反應等）

**我會做的事：**
1. 分析問題
2. 修復「發布阻擋」級別的 bug
3. 告訴你需要測試什麼

**驗收標準：**
- [ ] 所有頁面可以正常載入

- [ ] 沒有 API 401/500 錯誤

## 😴 休息期（1 週）

### 2 月 1 日 - 2 月 7 日

**規則：**
- 不主動開發新功能
- 只處理「客戶無法使用」級別的緊急 bug
- 每天最多花 30 分鐘檢查系統狀態
- 做你想做的其他事情

**為什麼需要休息？**
- 兩個多月的高強度開發需要恢復
- 讓客戶先用一週，收集真實反饋
- 有了真實反饋，才知道什麼是真正重要的

## 📋 每日工作流程

### 你的工作流程

```
09:00 - 09:30  檢查系統狀態（開啟各頁面，看有無問題）
09:30 - 10:00  把問題清單發給 Claude
10:00 - 11:00  等待 Claude 修復，做其他事情
11:00 - 11:30  測試修復結果
11:30 - 12:00  確認 OK 或回報新問題
```

**總計：2-3 小時/天**

### 與 Claude 的協作流程

```
1. 你發現問題或需求
   ↓
2. 用簡短的文字描述
   例：「搜尋框沒反應」
   ↓
3. Claude 使用 /debug-buygo 診斷
   ↓
4. Claude 說明修復計畫
   ↓
5. 你確認 OK
   ↓
6. Claude 修改代碼
   ↓
7. Claude 給你測試清單
   ↓
8. 你測試並回報結果
   ↓
9. OK → 進入下一個任務
   NOT OK → 回到步驟 4
```

### 問題回報格式

```
【問題】搜尋框沒反應
【頁面】商品管理
【步驟】輸入文字，按 Enter
【預期】列表應該過濾
```

```
【測試項目】搜尋功能
【結果】✅ OK
【備註】無
```

### 發現新問題格式

```
【測試項目】搜尋功能
【結果】❌ 有問題
【問題描述】輸入後有過濾，但清空搜尋框後列表沒有恢復
```

## ⚠️ 重要提醒

### 什麼時候找 Claude

```
✅ 應該找：
- 發現 bug
- 需要修改代碼
- 需要新功能
- 需要解釋代碼行為

❌ 不應該找：
- 可以自己測試的事情
- 需要等待的事情（部署、客戶反饋）
```

### 什麼時候暫停

```
如果你感到：
- 很累
- 很煩躁
- 一直在重複同樣的問題

請暫停，告訴 Claude：
「我需要休息一下，我們明天繼續」

這不是放棄，這是保護自己。
```

### 什麼時候調整計畫

```
計畫可以調整，如果：
- 某個任務比預期複雜
- 客戶有緊急需求
- 你需要更多休息時間

告訴 Claude，我們一起調整。
```
