<task type="auto">
  <name>Task 1: Create DataManagement_API with all REST endpoints</name>
  <files>buygo-plus-one/includes/api/class-data-management-api.php</files>
  <action>
Create `includes/api/class-data-management-api.php` with namespace `BuyGoPlus\Api`.

Follow the exact same class structure as Products_API and Orders_API:
- Private `$namespace = 'buygo-plus-one/v1'`
- Public `register_routes()` method
- Callback methods for each route
- Static `check_permission_for_admin()` that returns `current_user_can('buygo_admin') || current_user_can('manage_options')` (same pattern as Settings_API line 730-735)

Register these routes:

1. **GET /data-management/query**
   - permission_callback: check_permission_for_admin
   - args:
     - `type` (required): validate in ['orders', 'products', 'customers']
     - `date_from` (optional): sanitize_text_field, validate Y-m-d format
     - `date_to` (optional): sanitize_text_field, validate Y-m-d format
     - `keyword` (optional, default ''): sanitize_text_field
     - `page` (default 1): absint
     - `per_page` (default 20): absint
   - Callback: Instantiate `DataManagementService`, call `queryOrders()`, `queryProducts()`, or `queryCustomers()` based on `type` param
   - Return: WP_REST_Response with the service result

2. **POST /data-management/delete-orders**
   - permission_callback: check_permission_for_admin
   - args:
     - `ids` (required): validate is_array and !empty
     - `confirmation_token` (required): validate equals 'DELETE' exactly
   - Callback: Validate confirmation_token === 'DELETE', then call `DataManagementService::deleteOrders($ids)`
   - Return 400 if confirmation_token !== 'DELETE' with message '需要輸入 DELETE 確認刪除'

3. **POST /data-management/delete-products**
   - permission_callback: check_permission_for_admin
   - args: same as delete-orders (ids + confirmation_token)
   - Callback: Validate confirmation_token, call `DataManagementService::deleteProducts($ids)`

4. **POST /data-management/delete-customers**
   - permission_callback: check_permission_for_admin
   - args: same as delete-orders (ids + confirmation_token)
   - Callback: Validate confirmation_token, call `DataManagementService::deleteCustomers($ids)`

5. **PUT /data-management/customers/{id}**
   - permission_callback: check_permission_for_admin
   - args:
     - `id` (required): validate is_numeric
     - `first_name`, `last_name`, `phone`, `address_1`, `address_2`, `city`, `state`, `postcode`, `country`: all sanitize_text_field
     - `taiwan_id_number`: sanitize_text_field
   - Callback: Call `DataManagementService::updateCustomer($id, $data)`
   - Return 404 if customer not found, 200 on success

IMPORTANT for DATA-05 (confirmation token):
All three delete endpoints MUST check `confirmation_token === 'DELETE'` in the callback body BEFORE calling the service. This is the server-side enforcement of the two-step confirmation. The first step (showing count Modal) is frontend-only. The second step (typing DELETE) is enforced here.

Error handling: Wrap all callbacks in try/catch, return 500 with error message on exception (same pattern as other API classes).
